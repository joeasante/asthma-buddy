**3. Create `test/controllers/mfa_challenge_controller_test.rb`:**

```ruby
class MfaChallengeControllerTest < ActionDispatch::IntegrationTest
```

Setup: `@mfa_user = users(:mfa_user)` and `@otp_secret = "JBSWY3DPEHPK3PXP"` (must match fixture).

Helper method to enter pending MFA state:
```ruby
def enter_pending_mfa_state(user)
  post session_path, params: { email_address: user.email_address, password: "password123" }
  # Should redirect to MFA challenge
  assert_redirected_to new_mfa_challenge_path
end
```

Tests:

- `test "GET /mfa-challenge without pending state redirects to login"`:
  GET `new_mfa_challenge_path` without posting credentials first. Assert redirected to `new_session_path`.

- `test "GET /mfa-challenge with pending state renders form"`:
  `enter_pending_mfa_state(@mfa_user)`. GET `new_mfa_challenge_path`. Assert response 200. Assert `assert_select "input[name=otp_code]"`.

- `test "POST /mfa-challenge with valid TOTP code authenticates user"`:
  `enter_pending_mfa_state(@mfa_user)`.
  Generate valid code: `code = ROTP::TOTP.new(@otp_secret, issuer: "Asthma Buddy").now`.
  `assert_difference "Session.count", 1 do post mfa_challenge_path, params: { otp_code: code } end`.
  Assert redirected (to root or return URL). Assert `session[:pending_mfa_user_id]` is nil (cleared).

- `test "POST /mfa-challenge with invalid TOTP code re-renders form"`:
  `enter_pending_mfa_state(@mfa_user)`.
  `post mfa_challenge_path, params: { otp_code: "000000" }`.
  Assert response 422. Assert `assert_select "input[name=otp_code]"` (form re-rendered).

- `test "POST /mfa-challenge with valid recovery code authenticates user"`:
  `enter_pending_mfa_state(@mfa_user)`.
  `assert_difference "Session.count", 1 do post mfa_challenge_path, params: { otp_code: "testcode001" } end`.
  Assert redirected. Assert flash notice mentions "Recovery code used". Assert `@mfa_user.reload.recovery_codes_remaining` decreased by 1.

- `test "POST /mfa-challenge with already-used recovery code fails"`:
  `enter_pending_mfa_state(@mfa_user)`.
  Use recovery code "testcode001". Follow redirect (authenticated).
  Sign out. Re-enter pending MFA state.
  `post mfa_challenge_path, params: { otp_code: "testcode001" }`.
  Assert response 422 (code already consumed).

- `test "POST /mfa-challenge with expired pending state redirects to login"`:
  `enter_pending_mfa_state(@mfa_user)`.
  Manually set session expiry: this is tricky in integration tests. Alternative approach: use `travel_to 6.minutes.from_now` (Rails time helper), then POST with a valid code. Assert redirected to `new_session_path` (expired).

- `test "POST /mfa-challenge without pending state redirects to login"`:
  POST `mfa_challenge_path, params: { otp_code: "123456" }` without entering pending state. Assert redirected to `new_session_path`.

Note on time travel for expiry test: Use `travel_to` or `travel` from `ActiveSupport::Testing::TimeHelpers` (included by default in Rails tests). Generate the TOTP code BEFORE time travel (since TOTP is time-based, traveling 6 minutes would change the valid code). The test should verify the PENDING STATE expiry (5 min), not the TOTP code expiry.

**MFA Setup Tests:**

- `test "GET /settings/security shows MFA status disabled"`:
  Sign in as verified_user. GET `settings_security_path`. Assert 200. Assert page shows enable button/link (assert_select for the setup link).

- `test "GET /settings/security shows MFA status enabled"`:
  Sign in as mfa_user. GET `settings_security_path`. Assert 200. Assert page shows enabled status and disable link.

- `test "GET /settings/security/setup renders QR code"`:
  Sign in as verified_user. GET `setup_settings_security_path`. Assert 200. Assert page contains SVG (assert_select "svg" or check for QR-related content). Assert session[:pending_otp_secret] is set.

- `test "POST /settings/security/confirm_setup with valid code enables MFA"`:
  Sign in as verified_user. GET `setup_settings_security_path` (to set pending secret in session).
  Extract the pending secret from the session (in integration tests, use the session hash).
  Generate valid code: `code = ROTP::TOTP.new(session_secret, issuer: "Asthma Buddy").now`.
  POST `confirm_setup_settings_security_path, params: { otp_code: code }`.
  Assert redirected to `recovery_codes_settings_security_path`.
  Assert `users(:verified_user).reload.otp_required_for_login?` is true.

  NOTE: Accessing session in integration tests — the session is available via the `session` method after a request. However, `session[:pending_otp_secret]` is set by the controller during the GET. To get a valid code, you may need to read the secret from the session after the GET request, then generate a TOTP code from it. If session access is limited, an alternative approach: set a known secret directly by using `ROTP::Base32.random` in the test, stub/intercept, or simply test the full flow by making the GET request, extracting the QR page content for the manual key (displayed as text on the page), and using that to generate a code. Prefer the approach that works with the testing framework.

- `test "POST /settings/security/confirm_setup with invalid code re-renders setup"`:
  Sign in as verified_user. GET `setup_settings_security_path`.
  POST `confirm_setup_settings_security_path, params: { otp_code: "000000" }`.
  Assert response 422. Assert `users(:verified_user).reload.otp_required_for_login?` is false (MFA not enabled).

- `test "POST /settings/security/confirm_setup without pending secret redirects"`:
