**2. controller create action — compute @severity_counts after a successful save.**

In the `create` action, inside the `if @symptom_log.save` branch, before the `respond_to` block, add:

```ruby
@severity_counts = { mild: 0, moderate: 0, severe: 0 }.merge(
  Current.user.symptom_logs.severity_counts
)
```

This computes the full-history counts (no date filter) which is what the trend bar shows on a fresh page load. Do not add date filtering here — the trend bar always reflects the user's complete history, not the current filter window.

Place this as the first operation in the file (before the prepend) so the bar updates in the same render cycle as the new row appearing. The rendered inner content must include the `id="trend_bar"` div itself because `turbo_stream.replace` replaces the target element wholesale including its wrapper.

Three coordinated changes plus test additions.

**1. index.html.erb — move filter_bar render inside the timeline_content Turbo Frame.**

Currently the filter_bar is rendered OUTSIDE the `turbo_frame_tag "timeline_content"` block. This was the original Phase 5 Plan 01 decision. That decision must now be reversed: the active-chip CSS class is set by the partial using `active_preset` which is only re-rendered when the frame refreshes. Moving the filter_bar inside the frame means every chip click and date submit re-renders the bar with the correct active chip.

Find the section that looks like:

```erb
<%= render "filter_bar",
      active_preset: @active_preset,
      start_date: @start_date,
      end_date: @end_date %>

<%= turbo_frame_tag "timeline_content" do %>
  ...
<% end %>
```

Change it so the filter_bar render is the FIRST item inside `turbo_frame_tag "timeline_content"`:

```erb
<%= turbo_frame_tag "timeline_content" do %>
  <%= render "filter_bar",
        active_preset: @active_preset,
        start_date: @start_date,
        end_date: @end_date %>

  <div id="trend_bar">
    <%= render "trend_bar", severity_counts: @severity_counts %>
  </div>
  ...rest of frame content unchanged...
<% end %>
```

The `_filter_bar.html.erb` partial itself does NOT need to change. The chip links use `data: { turbo_frame: "timeline_content" }` — when the bar is inside the frame these links navigate the frame and the frame re-renders including the bar with the updated active chip.

Note: STATE.md records "Filter bar sits OUTSIDE turbo_frame_tag 'timeline_content'" as a prior decision. That decision is being superseded here by this gap fix. The root cause of the 30d chip appearing broken was that the active-chip visual state never updated because the bar was outside the frame. Moving it inside is the correct fix.

**2. _form.html.erb — add step: 60 to the datetime_local_field.**

Change:

```erb
<%= form.datetime_local_field :recorded_at %>
```

To:

```erb
<%= form.datetime_local_field :recorded_at, step: 60 %>
```

This sets the HTML `step` attribute to 60 seconds, aligning the browser's valid-value anchor to whole minutes.

**3. controller index action — change Time.current to Time.current.change(sec: 0).**

In the `index` action's `format.html` branch:

```ruby
format.html { @symptom_log = Current.user.symptom_logs.new(recorded_at: Time.current) }
```

Change to:

```ruby
format.html { @symptom_log = Current.user.symptom_logs.new(recorded_at: Time.current.change(sec: 0)) }
```

This ensures the default pre-filled value in the form is already at a whole-minute boundary, so the browser's step-60 validation never rejects it on initial load or when the form is pre-populated.
