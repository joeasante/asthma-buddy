<objective>
Add two complementary security controls: IP-level rate limiting via rack-attack (guards the login and signup endpoints against brute-force) and idle session timeout via a before_action (terminates abandoned authenticated sessions after 60 minutes of inactivity).

Purpose: Reduce the blast radius of credential-stuffing attacks and limit the exposure window of unattended sessions in a HIPAA-adjacent health app.
Output: rack_attack.rb initializer, middleware registration, check_session_freshness before_action, and integration tests proving both controls fire correctly.
</objective>

    1. Add `gem "rack-attack"` to Gemfile (in the main gem group, after `gem "bcrypt"`). Run `bundle install`.

    2. Create `config/initializers/rack_attack.rb`:

    ```ruby
    # frozen_string_literal: true

    class Rack::Attack
      # Throttle login attempts: 5 per IP per 20 seconds
      throttle("logins/ip", limit: 5, period: 20) do |req|
        req.ip if req.path == "/session" && req.post?
      end

      # Throttle signup attempts: 3 per IP per hour
      throttle("signups/ip", limit: 3, period: 1.hour) do |req|
        req.ip if req.path == "/registration" && req.post?
      end

      # Custom response for throttled requests
      self.throttled_responder = lambda do |_env|
        [429, { "Content-Type" => "text/plain" }, ["Too many requests. Please wait before trying again."]]
      end
    end
    ```

    IMPORTANT — path correction: Routes are `resource :session` and `resource :registration` (both singular resources). The correct paths are `/session` and `/registration`, NOT `/sessions` or `/registrations`. Do not use the plural forms.

    3. In `config/application.rb`, inside `class Application < Rails::Application`, add after the existing `config.action_dispatch.default_headers.merge!(...)` block:

    ```ruby
    config.middleware.use Rack::Attack
    ```

    4. Disable rack-attack in the test environment to prevent interference with unrelated tests. Create or append to `config/environments/test.rb`:

    ```ruby
    # Disable rack-attack globally in tests; integration tests re-enable it selectively.
    config.middleware.delete Rack::Attack
    ```

    If `config/environments/test.rb` already has content, append this before the final `end`.

  <files>

  <action>
    1. In `app/controllers/application_controller.rb`, add a `before_action :check_session_freshness` and the private method. Place the before_action AFTER the existing before_actions (allow_browser, set_no_store_cache_for_authenticated_users, set_notification_badge_count).

    The constant and method go in the private section:

    ```ruby
    IDLE_TIMEOUT = 60.minutes

    before_action :check_session_freshness

    private

    def check_session_freshness
      return unless session[:last_seen_at]
      if Time.current - session[:last_seen_at].to_time > IDLE_TIMEOUT
        reset_session
        redirect_to new_session_path, alert: "Your session expired due to inactivity. Please sign in again."
      else
        session[:last_seen_at] = Time.current
      end
    end
    ```

    2. Add `skip_before_action :check_session_freshness` to every controller that handles unauthenticated access. These controllers must NOT have the timeout check run on them:

    - `SessionsController` — add `skip_before_action :check_session_freshness` (already has `allow_unauthenticated_access`)
    - `RegistrationsController` — same
    - `PasswordsController` — same
    - `EmailVerificationsController` — same
    - `PagesController` — public legal pages
    - `CspReportsController` — browser-initiated, no session
    - `ErrorsController` — error pages served unauthenticated
    - `CookieNoticesController` — public dismissal action
    - `HomeController` — root path, public
    - `Test::SessionsController` — test-only, unauthenticated

    For each controller, add `skip_before_action :check_session_freshness` immediately after any existing `allow_unauthenticated_access` or `before_action` declarations at the top of the class body. Do not disturb existing controller logic.

    3. In `app/controllers/sessions_controller.rb`, set `session[:last_seen_at]` immediately after `start_new_session_for user`, before the `respond_to` block:

    ```ruby
    start_new_session_for user
    session[:last_seen_at] = Time.current
    ```

    This ensures the freshness timestamp is written at login time so the first authenticated request after login does not trigger a false timeout.

    NOTE: Do NOT set `last_seen_at` in the `Test::SessionsController` — that controller is only for signing in via fixture sessions in system tests, and the timeout tests must control `last_seen_at` directly in their setup.
  </action>

    ```ruby
    # frozen_string_literal: true

    require "test_helper"

    class RateLimitingTest < ActionDispatch::IntegrationTest
      setup do
        # Re-enable rack-attack for this test class and reset its counters.
        Rack::Attack.enabled = true
        Rack::Attack.reset!
      end

      teardown do
        Rack::Attack.enabled = false
        Rack::Attack.reset!
      end

      test "6th login POST from same IP within 20 seconds is throttled with 429" do
        5.times do
          post session_path, params: { email_address: "x@example.com", password: "wrong" },
            headers: { "REMOTE_ADDR" => "1.2.3.4" }
        end
        post session_path, params: { email_address: "x@example.com", password: "wrong" },
          headers: { "REMOTE_ADDR" => "1.2.3.4" }
        assert_response 429
      end

      test "different IPs are not affected by each other's login throttle" do
        5.times do
          post session_path, params: { email_address: "x@example.com", password: "wrong" },
            headers: { "REMOTE_ADDR" => "1.2.3.4" }
        end
        post session_path, params: { email_address: "x@example.com", password: "wrong" },
          headers: { "REMOTE_ADDR" => "9.9.9.9" }
        assert_response :redirect  # redirected to login with alert — not 429
      end

      test "4th signup POST from same IP within one hour is throttled with 429" do
        3.times do
          post registration_path,
            params: { user: { email_address: "a@example.com", password: "pass1234", password_confirmation: "pass1234" } },
            headers: { "REMOTE_ADDR" => "2.3.4.5" }
        end
        post registration_path,
          params: { user: { email_address: "b@example.com", password: "pass1234", password_confirmation: "pass1234" } },
          headers: { "REMOTE_ADDR" => "2.3.4.5" }
        assert_response 429
      end
    end
    ```

    ```ruby
    require "test_helper"

    class SessionTimeoutTest < ActionDispatch::IntegrationTest
      setup do
        @user = users(:regular_user)
      end

      test "session with last_seen_at 61 minutes ago is expired and redirects to login" do
        sign_in_as(@user)
        # Backdate the session timestamp to simulate 61 minutes of inactivity.
        session[:last_seen_at] = 61.minutes.ago
        get dashboard_path
        assert_redirected_to new_session_path
        assert_equal "Your session expired due to inactivity. Please sign in again.", flash[:alert]
      end

      test "session with last_seen_at 59 minutes ago passes through normally" do
        sign_in_as(@user)
        session[:last_seen_at] = 59.minutes.ago
        get dashboard_path
        assert_response :success
      end

      test "last_seen_at is updated on each authenticated request" do
        sign_in_as(@user)
        session[:last_seen_at] = 30.minutes.ago
        get dashboard_path
        assert_response :success
        # Timestamp must have been refreshed to approximately now.
        assert_in_delta Time.current.to_i, session[:last_seen_at].to_time.to_i, 5
      end

      test "session without last_seen_at set passes through (backward compatibility)" do
        sign_in_as(@user)
        session.delete(:last_seen_at)
        get dashboard_path
        assert_response :success
      end
    end
    ```
