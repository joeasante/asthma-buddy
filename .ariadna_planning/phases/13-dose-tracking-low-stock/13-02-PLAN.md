**Routes** (`config/routes.rb`):

Update the settings scope to add a `refill` member action on medications:

```ruby
scope "/settings", module: :settings, as: :settings do
  resources :medications, only: %i[index new create edit update destroy] do
    member do
      patch :refill
    end
    resources :dose_logs, only: %i[create destroy]
  end
end
```

This generates `PATCH /settings/medications/:id/refill` → `settings#medications#refill`, named `refill_settings_medication_path(medication)` (or `settings_medication_refill_path(medication)` — verify with `bin/rails routes | grep refill` after adding).

**Controller** (`app/controllers/settings/medications_controller.rb`):

Add `refill` to the `before_action :set_medication` list: `before_action :set_medication, only: %i[edit update destroy refill]`.

Add the `refill` action after `destroy`:

```ruby
def refill
  new_count = refill_params[:starting_dose_count].to_i
  if new_count >= 0 && @medication.update(starting_dose_count: new_count, refilled_at: Time.current)
    flash.now[:notice] = "#{@medication.name} refilled. #{new_count} doses recorded."
    respond_to do |format|
      format.turbo_stream
      format.html { redirect_to settings_medications_path, notice: flash.now[:notice] }
    end
  else
    flash.now[:alert] = "Refill count must be 0 or greater."
    respond_to do |format|
      format.turbo_stream { render turbo_stream: turbo_stream.replace("flash-messages", partial: "layouts/flash"), status: :unprocessable_entity }
      format.html { redirect_to settings_medications_path, alert: flash.now[:alert] }
    end
  end
end
```

Add private method `refill_params`:

```ruby
def refill_params
  params.require(:medication).permit(:starting_dose_count)
end
```

Authorization is handled by `set_medication` which uses `Current.user.medications.find` — cross-user PATCH returns 404 automatically.

Important: `flash.now[:notice]` is set before `respond_to` so Turbo Stream views that render the flash partial receive the value. This follows the established pattern from Phase 12 Plan 02.

Wait — the Turbo Frame pattern for toggle: clicking "Refill" should load the refill form into the same frame without a separate `edit_refill` route. The simpler approach (per CONTEXT.md: "Turbo Frame toggles between Refill button and a small inline form") is to embed the form directly in the card and use a Stimulus controller or — simpler — a second Turbo Frame pointing to the same page:

Use a dedicated approach: render both the button and the form in the partial, hide the form by default with CSS (`display: none`), and use a Stimulus `toggle` controller. But per CONTEXT.md, "Claude's Discretion" allows choosing Turbo Frame or Stimulus.

Recommended implementation (Turbo Frame approach, no extra route needed):

Embed the refill form inline, hidden inside a turbo_frame. Add a second Turbo Frame tag with `src` pointing to a `?show_refill=1` variant OR — simplest approach — just render the form always visible but collapsible via a `<details>` HTML element:

```erb
<details class="refill-details">
  <summary class="btn-refill">Refill</summary>
  <%= form_with url: refill_settings_medication_path(medication), method: :patch,
        data: { turbo_frame: dom_id(medication) } do |f| %>
    <div class="refill-form">
      <%= f.label :starting_dose_count, "New dose count" %>
      <%= f.number_field :starting_dose_count,
            value: medication.starting_dose_count,
            min: 0,
            class: "refill-count-input",
            name: "medication[starting_dose_count]" %>
      <%= f.submit "Confirm refill", class: "btn-primary btn-sm" %>
    </div>
  <% end %>
</details>
```

The `<details>/<summary>` pattern requires zero JavaScript, is accessible, and matches "one-click with editable default" from the locked decision: summary acts as the toggle button; the form inside is pre-filled with `medication.starting_dose_count`; user either clicks "Confirm refill" (one click after opening) or adjusts the number first.

1. `remaining_count_{dom_id(@medication)}` — refreshes the remaining dose count (now based on new `starting_dose_count`), days-of-supply text, and low-stock badge
2. `flash-messages` — shows the success notice
3. Optionally the whole medication Turbo Frame `dom_id(@medication)` to reset the `<details>` element to closed state

Best approach: replace the entire medication card turbo_frame (dom_id(@medication)) to get a fully fresh card with updated counts and closed refill form, plus replace flash:

```erb
<%= turbo_stream.replace dom_id(@medication) do %>
  <%= render "settings/medications/medication", medication: @medication %>
<% end %>

<%= turbo_stream.replace "flash-messages" do %>
  <%= render "layouts/flash" %>
<% end %>
```

This approach fully re-renders the medication card (including updated remaining_doses, low_stock? badge, days_of_supply_remaining) and resets the `<details>` element to closed. It is consistent with the Phase 11 Turbo Stream update pattern.

Important: `@medication` must be the reloaded record (fresh from DB) so `remaining_doses` recalculates against the new `starting_dose_count`. In the controller, after `@medication.update(...)` succeeds, `@medication` is already updated in memory — no explicit reload needed because `update` mutates the object in place.
