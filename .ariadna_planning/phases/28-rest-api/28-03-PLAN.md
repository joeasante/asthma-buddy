  <action>
    1. Update `config/initializers/rack_attack.rb` to add API-specific throttle:

       Add a new throttle rule for API requests:
       ```ruby
       throttle("api/v1/requests", limit: 60, period: 1.minute) do |req|
         if req.path.start_with?("/api/v1/")
           # Throttle by API key digest (extracted from Bearer token)
           token = req.env["HTTP_AUTHORIZATION"]&.match(/\ABearer (.+)\z/)&.[](1)
           Digest::SHA256.hexdigest(token) if token.present?
         end
       end
       ```
       This ensures:
       - Only /api/v1/ paths are throttled by this rule
       - Throttle key is the SHA-256 digest of the token (same as stored in DB)
       - Web requests (no Bearer token, different paths) are completely unaffected
       - Different API keys have independent rate counters

    2. Update the `throttled_responder` to handle API throttle responses:
       - When `rack.attack.matched` is "api/v1/requests":
         - Return 429 status
         - Include `Retry-After` header (use the throttle period in seconds: 60)
         - Body: `{ error: { status: 429, message: "Rate limit exceeded. Try again later.", details: nil } }` — matching the consistent error format from the API controllers
         - Content-Type: application/json
       - Compute Retry-After from `rack.attack.match_data`: `match_data[:period] - (Time.now.to_i % match_data[:period])` gives seconds until the throttle window resets

    3. Create `test/controllers/api/v1/rate_limiting_test.rb`:
       - In setup: enable Rack::Attack (`Rack::Attack.enabled = true`), reset Rack::Attack cache
       - In teardown: disable Rack::Attack, reset cache
       - Generate an API key for the test user

       Tests:
       - **API requests within limit succeed:** Make 3 requests, all return 200
       - **API requests exceeding limit get 429:** Make 61 requests in a loop, verify the last one returns 429
       - **429 response includes Retry-After header:** Check `response.headers["Retry-After"]` is present and is a positive integer
       - **429 response body is consistent JSON error format:** Parse body, verify `{ error: { status: 429, message: "..." } }` structure
       - **Web requests are not affected by API rate limit:** After exhausting API limit, make a regular web GET request (e.g., GET /settings with session cookie) and verify it returns 200 (not 429)
       - **Different API keys have independent limits:** User A exhausts limit, User B can still make requests

       Note: The existing test pattern for Rack::Attack (test/controllers/rate_limiting_test.rb) enables/disables Rack::Attack in setup/teardown. Follow that same pattern. Use `Rack::Attack.cache.store.clear` (or `Rack::Attack.reset!`) to reset counters between tests.

    4. Run `bin/rails test test/controllers/api/v1/rate_limiting_test.rb` and then full suite.
